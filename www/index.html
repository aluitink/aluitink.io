Ojibwe
Encrypted Cloud Storage Abstraction Layer

I've been dreaming this one up for a while. Ojibwe is the first step in a two part project that's goal is to provide a powerful encrypted cloud storage solution that is scalable. The project uses both symmetrical and asymmetrical encryption to provide fast and secure cloud blob storage. This solution only puts encrypted bytes over the wire. An implementer will use the SDK to create containers and blobs. When a blob is stored, a random symmetrical key is created, the data is encrypted, the key is then encrypted with the users public key and stored in the key storage database. The blob is then uploaded and stored on the servers file system or in AmazonS3. The SDK has interfaces for new Storage Adapters and Data Access (schema, key, and blob storage system). Blobs can be shared, keys are shared by re-encrypting them with destination users public key.

Navajo - Phase 2, includes layering Document Storage and indexing on top of this to allow for more interesting applications to be developed. I imagine a social network where the network provider has no visibility into users information, and a user could sell their info to the network for a fee.

ManagedMapi
.NET Mapi Provider

My work on the Archive Manager product gave me a lot of experience with Microsoft's Mail API (MAPI). MAPI is an interesting library because of how it provides data to you. MAPI is a collection of native interfaces that you can query to access mail on an Exchange system.

There are various flavors of MAPI, but they are more or less called the same. The client side of the connection allows you to talk to a MAPI Store Provider and communicate with something on the other end. There is a managed client implementation called MAPI33 that seems to have been removed from the Internet. This allowed the query of the MAPI subsystem from Managed code. The only missing part was a MAPI provider implementation that allows you to create a provider for MAPI.

We had this idea of allowing users to access their Archived Email directly from Outlook without needing to write an Outlook Add-in. If we had a MAPI provider, we could simply add it to the users mail profile and let Outlook display it all.

Our local MAPI expert wrote most of the ground work, dealing with all of the marshalling from Native to Managed code. I started work on the Managed provider implementation and the Native facade used to invoke the managed implementation. We never officially released this but it was a great learning experience on how to invoke Managed code via COM from unmanaged code.
